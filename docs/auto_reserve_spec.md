# 超星图书馆座位自动预约脚本需求说明书

## 1. 背景与目标

本项目相关的原始数据为某高校图书馆基于超星「座位预约」系统的使用记录与抓包数据，存放在本仓库中：

- 规则页面 HTML：`rules.txt`
- 图书馆官方预约规则 PDF：`规则.pdf`
- 抓包原始数据目录：`2026-03-04-231350/office.chaoxing.com/**`

当前系统支持在 Web 端（Chrome 浏览器）进行座位预约，包括「预约选座」「快速选座」等功能。用户希望在遵守图书馆预约规则的前提下，减少每天到点手动抢座的重复操作。

**本项目的目标：**

实现一个基于 Python 的「单次执行」脚本，用于在预约开放时间附近自动尝试预约用户预设的一组座位。定时调度由���作系统层面的定时任务负责（如 Linux `cron`、Windows 任务计划等），脚本本身只负责执行“一轮预约尝试”。

> 备注：本脚本仅面向单一用户账号使用，不用于刷号、批量抢占资源或任何形式的攻击性行为。

## 2. 系统与业务规则概述

本节对图书馆座位系统的主要规则进行抽象描述，用于指导脚本设计。具体数值来源于本仓库中的 `规则.pdf` 与 `rules.txt`。

### 2.1 使用流程（业务流）

官方规则定义的座位使用流程为：

1. **预约座位**
   - 预约选座：
     - 在首页点击「预约选座」按钮；
     - 选择需要预约的时间、区域、座位；
     - 适用于有明确目标座位的情况。
   - 快速选座：
     - 在首页点击「快速选座」按钮；
     - 设置筛选条件，由系统自动分配满足条件的座位；
     - 适用于对具体座位不敏感的情况。

2. **签到**
   - 使用手机扫描桌子上的二维码，在签到范围内可以完成签到；
   - 签到时间窗口：**预约开始时间后的 30 分钟内**；
   - 若未在规定时间内签到，将被记录为 **一次违约**。

3. **暂离**
   - 如需暂时离开座位，应在首页当前预约卡片中点击「暂离」按钮；
   - 暂离时长：**30 分钟**；
   - 在暂离时间内返回并扫码落座，座位将保持占用状态；
   - 逾期未返回并落座，将被记录为违约。

4. **退座**
   - 使用完毕后，系统默认座位会自动退座；
   - 如果提前较长时间离开，应主动点击「退座」按钮；
   - 退座后座位释放，同时记录 **一次成功履约**。

### 2.2 通用预约规则（普通区域）

根据当前规则：

1. **预约开放时间**  
   - 预约开始时间为：**提前 1 天 的 20:00**  
   - 示例：  
     - 想预约「3 月 6 日」的座位，需要在「3 月 5 日 20:00」之后发起预约。

2. **单次预约时长**  
   - 普通区域：**不限制单次预约时长**（由用户在界面中选择）；

3. **座位预留时长（签到窗口）**  
   - 座位预留时长为：**30 分钟**（预约开始后 30 分钟内必须完成签到）��

4. **取消次数限制**  
   - 每日取消次数：**不限**；
   - 每周取消次数：**3 次**。

5. **暂离与监督**  
   - 暂离时长：**30 分钟**；
   - 监督时长：**10 分钟**（被监督后需要在 10 分钟内处理）。

### 2.3 特殊预约规则（特殊区域）

存在“特殊区域”（例如：文北校区 - 3楼 - 学生研讨间），其规则与普通区域略有不同：

- 特殊区域：
  - 单次预约时长：**2 小时**；
  - 座位预留时长：**30 分钟**；
  - 暂离时长：**30 分钟**；
  - 监督时长：**10 分钟**。

> v1 版本的脚本以「普通自习区域」为主要目标，但在设计时需保留对“预约时长”“区域/房间 ID”参数的扩展能力，以支持未来对特殊区域的自动预约。

### 2.4 监督规则概述

监督机制（简化）：

1. 任意读者发现某座位标记为“在使用”但无人就座，可通过扫描座位二维码发起监督；
2. 被监督人必须在 **10 分钟内**：
   - 重新扫码落座，继续使用座位；或
   - 主动退座，座位释放，记为正常履约；
3. 若在监督时间内未重新落座或退座，则被记录为一次违约，且座位释放。

### 2.5 违规与封禁规则（关键）

违规情形包括但不限于：

1. 预约后未在规定时间内（30 分钟）签到；
2. 离开不退座；
3. 暂离后 30 分钟内未返回扫码落座；
4. 被监督后 10 分钟内未返回或退座。

**封禁策略：**

- 每 **15 天** 内违约次数达到 **3 次**，将暂停 **7 天** 的预约功能。

> 对脚本的影响：脚本必须尽量避免触发异常行为，不应实现任何“自动作弊签到”“自动刷预约/取消”的逻辑，只辅助执行正常的预约操作。

## 3. 自动预约脚本 v1 目标与范围

### 3.1 v1 目标（必须实现）

1. 提供一个基于 Python 的脚本（例如：`reserve_once.py`），**执行一次**完整的“自动预约尝试”流程；
2. 支持通过配置文件指定：
   - 目标预约日期（默认为“明天”）；
   - 预约时间段（例如：08:00-12:00）；
   - 座位 ID 列表（有优先级顺序）；
   - 必要的区域/房间 ID 等参数；
3. 在执行时，根据配置构造请求，对目标座位依次尝试预约，一旦成功即停止继续尝试；
4. 将每次执行的尝试过程、结果、错误信息记录到日志文件或标准输出。

### 3.2 v1 非目标（暂不实现）

以下功能在 v1 中 **不要求** 实现，仅可在需求文档中作为“后续可能扩展”列出：

- 自动登录（模拟用户名密码输入、处理验证码、人机验证等）；
- 自动签到（自动扫码或模拟签到请求）；
- 自动退座或自动暂离操作；
- 图形化界面（GUI）或 Web 控制面板；
- 通知系统（如邮件、Push、微信/Telegram 机器人）。

## 5. 技术方案设计

### 5.1 语言与运行环境

- 语言：Python 3.x
- 运行环境：
  - 任意能够访问图书馆预约系统的环境：
    - 本地笔记本；
    - 寝室服务器（长期开机）；
  - 操作系统：
    - Linux / macOS / Windows 均可。

### 5.2 依赖建议

建议使用的 Python 库：

- HTTP 客户端：
  - `requests`
- 配置管理（可选）：
  - `pyyaml`（用于读取 `config.yml`）
  - `python-dotenv`（用于读取 `.env` 中的敏感信息，如 Cookie）
- 时间与日期处理：
  - 标准库 `datetime` 即可

> v1 可不引入定时任务库，因为定时由操作系统外部负责。

### 5.3 模块划分建议

建议（非强制）结构：

- `config/`
  - `config.yml`：非敏感的业务配置（目标日期、时间段、座位列表等）
- `.env`（不提交到仓库）：
  - Cookie、账号标识等敏感数据
- `reserve_once.py`（或 `main.py`）：
  - 程序入口，执行一轮自动预约逻辑
- `lib/`（可选）：
  - `http_client.py`：封装请求发送、重试逻辑
  - `reservation.py`：构造预约请求、处理结果
  - `logging_utils.py`：简单日志封装

## 6. 配置设计

### 6.1 config.yml（示例草案）

```yaml
# config/config.yml

# 目标日期模式：
# - "tomorrow": 预约明天的座位（根据当前日期自动计算）
# - "date": 指定具体日期（配合 target_date 使用）
target_date_mode: "tomorrow"

# 当 target_date_mode 为 "date" 时使用，格式：YYYY-MM-DD
target_date: "2026-03-10"

# 预约时间段（示例：08:00-12:00）
time_range:
  start: "08:00"
  end: "12:00"

# 座位优先级列表（按顺序尝试）
seat_ids:
  - "SEAT_ID_1"
  - "SEAT_ID_2"
  - "SEAT_ID_3"

# 区域/房间等参数（具体字段名与值需基于抓包数据补充）
area:
  campus: "文北校区"
  building: "图书馆"
  floor: "二楼"
  room_id: "ROOM_ID_EXAMPLE"

# 运行行为参数
behavior:
  # 最多尝试的座位数量（防止过多请求）
  max_seats_to_try: 5

  # 每个座位之间的最小间隔（秒），避免过于频繁请求
  interval_seconds_between_seats: 0.2

# 日志配置
logging:
  level: "INFO"
  file: "reserve.log"
```

> 说明：上面的字段名/值仅为示例，实际具体字段（例如 `seat_id`、`room_id` 对应的字段名）需要参考 `2026-03-04-231350/office.chaoxing.com` 抓包中的请求 JSON 或表单字段进行调整。

## 7. ��约逻辑流程（v1）

### 7.1 高层流程（一次执行）

当通过系统定时任务在预约开放时间前后调用脚本时，脚本应执行如下步骤：

1. 读取配置：
   - 载入 `config/config.yml`；
   - 载入 `.env` 中的敏感信息（Cookie 等）。
2. 计算目标日期：
   - 若 `target_date_mode == "tomorrow"`，则目标日期为当前��期 + 1 天；
   - 若 `target_date_mode == "date"`，则直接使用 `target_date`。
3. 构造预约时间段：
   - 将目标日期与配置中的 `time_range.start`、`time_range.end` 组合为完整的开始/结束时间。
4. 迭代尝试座位列表：
   - 按 `seat_ids` 的顺序遍历，每次尝试一个座位：
     1. 根据抓包推导出的接口格式，构造一个预约请求；
     2. 设置必要的 HTTP 头和 Cookie；
     3. 发送请求并解析响应；
     4. 若返回成功（根据响应体中的状态码/字段判断），记录成功日志并终止流程；
     5. 若失败且失败原因不是“致命错误”（例如座位已被占用），则等待 `interval_seconds_between_seats` 秒后尝试下一个座位；
     6. 超过 `max_seats_to_try` 或座位列表耗尽，则停止尝试，记录最终失败。
5. 输出结果：
   - 在日志文件与控制台分别输出本次执行的概要结果（成功/失败、原因、座位 ID）。

### 7.2 错误与异常处理

- 如果接口返回认证失败（例如 Cookie 失效），脚本应：
  - 给出明确错误信息：“登录态失效，请重新在浏览器登录后更新 Cookie 配置”；
  - 不做任何进一步的自动重试。
- 如果接口返回参数错误，脚本应：
  - 打印调试信息（请求 URL、关键参数、响应片段）；
  - 提示开发者检查 `config.yml` 与抓包信息是否匹配。

## 8. 安全与合规要求

1. 本脚本仅用于辅助用户按照图书馆规定的流程进行预约，不应用于任何违反学校/系统使用协议的行为；
2. 禁止实现自动作弊签到、伪造地理位置、绕过验证码等“非正常手段”；
3. 禁止在公共仓库中提交账号、密码、完整 Cookie 等敏感信息；
4. 使用脚本过程中，如发现存在触发封禁风险的情况（例如频繁取消并重约），应在配置和逻辑中予以限制或提醒。

## 10. 验收标准（针对 v1）

完成 v1 开发后，应至少满足以下验收条件：

1. 在测试环境中（或非高峰期）通过抓包确认：
   - 脚本发送的预约请求结构与手动操作时的请求一致；
   - 在网络稳定且配置正确的情况下，能成功预约至少一个配置中的座位。

2. 在实际预约开放时间点：
   - 脚本被系统定时任务调用后能自动完成一次预约尝试；
   - 若指定的第一优先级座位已被占用，脚本会按顺序尝试后续座位；
   - 成功预约后终止后续尝试，并在日志中记录成功信息。

3. 在 Cookie 失效时：
   - 脚本能检测到异常（例如返回登录页面 HTML 等），并在日志中输出明确的错误提示，而不是无限重试。

4. 配置可维护性：
   - 用户无需修改源码，即可通过修改 `config/config.yml` 调整目标日期模式、时间段、座位列表；
   - `.env` 中的敏感信息不会被误提交到 Git 仓库。
