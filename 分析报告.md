## Cookie 结构分析

### Cookie 生成过程

```
↓ 用户访问座位系统首页
├─ (Request #1-3) 无 Cookie → 重定向到登录
├─ (Request #4) 登录成功 → 生成完整 Cookie 集合
└─ (Request #5+) 后续请求带上所有 Cookie
```

### Cookie 大小估计

单次请求的 Cookie 头大小: **~3-4 KB**

主要占用空间:
1. `vc3`: ~200+ 字节 (加密验证)
2. `uf`: ~200+ 字节 (用户指纹)
3. `p_auth_token`: ~150+ 字节 (JWT)

### Cookie 刷新策略

- 没有观察到 Cookie 版本更新
- 同一会话中 Cookie 值保持不变
- 建议实现自动刷新机制 (7 天有效期)

## 安全建议

### 🔴 高风险问题

1. **Cookie 中暴露未加密信息**
   ```
   _uid: 238771432
   UID: 238771432
   oa_name: 王红新 (URL 编码)
   ```
   建议: 所有敏感信息加密存储

2. **JWT Token 长期有效 (7 天)**
   - 风险: 令牌被盗后长期可用
   - 建议: 
     - 减少有效期到 1-2 小时
     - 实现刷新令牌机制
     - 添加令牌轮换

3. **JSESSIONID 暴露于 URL 中的可能性**
   - 建议: 强制 HTTPS + Secure/HttpOnly 标记

### 🟡 中等风险问题

1. **缺少明确的 CSRF 防护标记**
   - `cx_p_token` 可能是 CSRF token，但需要验证实现
   - 建议: 每个表单都应包含明确的 CSRF token

2. **分类层级信息泄露**
   - URL 中清晰暴露了建筑结构 (郑东校区 → 3楼 → 西阅览区)
   - 建议: 使用 ID 而非名称

3. **缺乏请求签名**
   - 修改参数可能能绕过验证
   - 建议: 对关键请求添加签名验证

### 🟢 良好实践

✓ 使用 HTTPS (推断)  
✓ 实现了 JWT 认证  
✓ 多层防护 (Cookie + JWT)  
✓ 使用了加密参数  
✓ 实现了防缓存机制  

### 建议改进措施

1. **增强 Token 管理**
   ```
   - 实现短期 access token (15 分钟)
   - 实现长期 refresh token (7 天)
   - 支持 Token 撤销/黑名单
   ```

2. **加强 Cookie 安全属性**
   ```
   Set-Cookie: p_auth_token=...; Secure; HttpOnly; SameSite=Strict
   ```

3. **实现请求签名验证**
   ```
   X-Signature: HMAC-SHA256(request_body, secret)
   ```

4. **完善审计日志**
   ```
   记录所有座位操作
   包含时间、用户、座位信息、IP 等
   ```

5. **实现速率限制**
   ```
   防止座位预约刷屏
   限制相同座位的重复请求
   ```
